# Design: Network Automation for Security — Firewall Rules as Code with Infrahub, Terraform & GitLab

**Date:** 2026-05-31
**Status:** Approved (design phase)
**Author:** pdamasceno (with Claude Code)

## Summary

A 5-part blog series (each part doubling as a video-walkthrough episode) that shows
platform/automation engineers how to manage **Cisco Firepower** firewall rules as code.
Infrahub is the source of truth (using the `experimental/security/security.yml` schema);
a GitLab CI pipeline runs Terraform that **reads** the intended policy from Infrahub and
**writes** Access Control rules to a Cisco Firepower Management Center (FMC). The custom
Infrahub Terraform provider is built from GraphQL queries and published to a **self-hosted
GitLab** registry.

## Goals

- Teach the end-to-end "firewall rules as code" loop for a security audience that already
  knows Terraform, CI/CD, and Git.
- Center of gravity: the Infrahub source-of-truth model, custom provider generation, and the
  dual-provider Terraform pipeline (including the gotchas).
- Be reproducible against **public Cisco DevNet/dCloud FMC sandbox** + a **self-hosted GitLab**.
- Each part ends with a working, demonstrable deliverable (so it maps cleanly to one episode).

## Non-Goals

- Not teaching firewall fundamentals or Terraform/Git basics from zero (audience is assumed fluent).
- Not building a custom FMC provider (we use the official `CiscoDevNet/fmc`).
- Not covering multi-vendor firewalls in this series (Firepower only; other vendors are a "what's next" teaser).

## Audience & Tone

Platform/automation engineers. Move fast on Terraform/CI/Git basics; go deep on the Infrahub
data model, provider code generation, the dual-provider pattern, state handling, and failure modes.

## Architecture

```
Engineer edits policy ─► Infrahub (security.yml schema = source of truth)
        │  branch + proposed change + merge
        ▼
   Infrahub webhook ─► GitLab CI pipeline
        │
        ▼
   Terraform run:
     • infrahub provider  (DATA SOURCE)  ← reads intended policy rules
         [provider built by us from GQL, served from self-hosted GitLab registry]
     • CiscoDevNet/fmc    (RESOURCE)     → writes Access Control rules to FMC
         [public Terraform registry]
     • state in GitLab-managed TF state (HTTP backend); plan posted back to the MR
        ▼
   Cisco Firepower (DevNet/dCloud sandbox FMC)
```

### Key technical facts (from exploration)

- The `marcom4rtinez/terraform-provider-infrahub` template generates **Terraform data sources
  from GraphQL queries** and **resources from GraphQL mutations**. Workflow: drop `.gql` files
  in `gql/`, set `INFRAHUB_SERVER`, run `make all` to generate SDK + provider + docs, then
  `make generate_deploy` (GoReleaser) to publish.
- The `security.yml` schema models: `SecurityPolicy` → `SecurityPolicyRule` (source/destination
  zones, addresses, address groups, services, service groups, action permit/deny, log) →
  `SecurityRenderedPolicyRule`; plus `SecurityFirewall` (inherits Dcim device generics) and
  `SecurityZone`/`SecurityFirewallInterface`.
- Live instance: `https://infrahub.autonetops.com` (token currently in untracked
  `schema-library/infrahubctl.toml` — see Security note).

### Per-part content outline

**Part 1 — The source of truth**
- Why a source-of-truth data model beats config templates for security policy.
- Tour of the `security.yml` schema (policy → rule → rendered rule; zones/addresses/services).
- Load the schema (`infrahubctl schema load`), model an example policy in the UI and as
  `infrahubctl` object files committed to `data/`.
- Deliverable: a populated example policy + reproducible `data/` files.

**Part 2 — Build your own Infrahub Terraform provider**
- How the template maps GQL queries → data sources, mutations → resources.
- Write the GQL query that returns the intended policy rules (see Validation V2).
- `make all`; inspect generated `internal/provider` + `docs/`.
- Retarget GoReleaser env to self-hosted GitLab; `make generate_deploy`; tag a release.
- Validate `terraform init` pulls the provider from GitLab.
- Deliverable: published provider + a `terraform/providers.tf` that resolves it. (See Validation V1.)

**Part 3 — Dual-provider Terraform, run locally**
- `providers.tf`: `infrahub` (our provider) + `CiscoDevNet/fmc`.
- Read the policy via the `infrahub_*` data source; map fields → FMC Access Control rule
  attributes (zones, networks, ports, action, logging).
- `terraform plan`/`apply` against the reserved DevNet/dCloud FMC sandbox; verify rules in FMC UI.
- Cover gotchas: FMC auth/domain UUID, rule ordering, object references, idempotency.
- Deliverable: a local Terraform run that lands rules on Firepower.

**Part 4 — The pipeline**
- `.gitlab-ci.yml`: `init` (pull infrahub provider from GitLab + fmc from public registry),
  `plan` (artifact + MR note), `apply` (on protected branch / manual gate).
- TF state via GitLab HTTP backend; secrets via masked/protected CI variables
  (`INFRAHUB_API_TOKEN`, FMC creds) — never committed.
- Infrahub webhook → GitLab pipeline trigger token on merge to `main`.
- Deliverable: a green pipeline that deploys on merge.

**Part 5 — Firewall rules as code (the payoff)**
- End-to-end day-2 demo on camera: branch in Infrahub → edit a rule → proposed change →
  review → merge → webhook → pipeline → rule on Firepower.
- Recap of the architecture; "what's next" teasers (other firewall vendors via additional
  providers, drift detection, rollback).
- Deliverable: the headline demo + series wrap.

## Validation Spikes (resolve, don't assume)

- **V1 — GitLab as a Terraform *provider* registry (Part 2).** GitLab's native registry targets
  Terraform *modules*. Confirm whether self-hosted GitLab can serve a *provider* via the registry
  protocol (GoReleaser endpoint retarget) or whether we must use the generic package registry +
  a minimal provider-registry index. Outcome documented; pick the working path before writing P2.
- **V2 — Rule rendering source (Part 1/3).** Decide between (a) an Infrahub **generator** that
  populates `SecurityRenderedPolicyRule` (groups/policies expanded server-side) and (b) querying
  `SecurityPolicyRule` directly and expanding in Terraform. Choose whichever keeps the data source
  cleanest and the GQL query simplest.

## Security Note (teaching thread)

The live API token currently sits in plaintext in untracked `schema-library/infrahubctl.toml`.
The series must model the correct pattern: secrets as GitLab **masked + protected** CI variables
(`INFRAHUB_API_TOKEN`, FMC username/password/host), referenced via env in Terraform/`infrahubctl`,
and never committed. We explicitly do *not* reproduce the plaintext-token pattern in published
artifacts. Recommend rotating the exposed token.

## Risks / Open Questions

- DevNet/dCloud FMC sandbox availability requires reservation; document the exact reservation
  steps and any session time limits in Part 3.
- `CiscoDevNet/fmc` provider version/feature coverage for the specific ACP rule attributes we
  map — pin a known-good version.
- Mapping fidelity between the Infrahub security model and FMC's object model (zones, network
  objects, port objects) may require creating supporting FMC objects in Terraform too.

## What "implementation" produces

Per part: the blog post markdown (in `infrahub-firewall-cicd/blog/`), the supporting artifacts
(GQL queries, `.tf` files, `.gitlab-ci.yml`, `data/` example objects, diagrams), and a
reproducibility checklist (exact commands + sandbox reservation steps). The video is the same
script performed, so each post doubles as a shot list.
